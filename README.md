# ⚖️ 勞資爭議數位證據鏈 Agent（Labor-Guard Agent）

黑客松模擬互動平台 — 使用 **Python + Streamlit** 打造，展示 AI Agent 蒐證解析 + Web3 區塊鏈存證 + 自動化調解申請書生成的完整概念驗證 (PoC)。

> ⚠️ 本專案為黑客松展示用途，區塊鏈上鏈與勞工局送件流程皆為**模擬**，非真實鏈上交易，生成之申請書亦非正式法律文件。

## 📌 專案簡介

打造全天候陪伴勞工的 **Labor-Guard Agent**，結合 AI 蒐證解析與 Web3 不可竄改證明，
讓「口說無憑」的勞資爭議，變成「有憑有據」的正式申訴。

### 🩹 核心痛點與現行狀況

- **舉證門檻高**：勞資爭議（如加班費未給付、違法解雇）時，勞工常面臨「口說無憑」的困境，且 LINE 對話紀錄、出勤截圖容易被資方質疑遭變造或刪除。
- **法規資訊不對等**：勞工往往不熟悉《勞動基準法》第 24 條（加班費）、第 38 條（特休）或《勞動事件法》相關舉證規定，導致協商時居於劣勢。
- **流程繁瑣**：撰寫勞資爭議調解申請書對一般勞工門檻極高，不知如何精準對應法條。
- **影響對象**：廣大受薪階級、中小企業勞工、非典型勞動者。

### 💡 AI Agent 解決方案

- **自動化蒐證與解析**：智慧解析勞工上傳的對話檔、班表，自動萃取關鍵時間、未給付薪資金額。
- **法規對齊**：自動匹配勞基法條，生成具有說服力的「事實與理由」。
- **Web3 信任根基**：將關鍵證據（檔案 SHA-256 雜湊）上鏈，確保資方無法否定其真實性，提供法庭或勞工局調解時的強力數位證據。
- **閉環服務**：從對話蒐證 → 區塊鏈存證 → 自動生成調解申請書 → 模擬送件，一站式完成。

### 🧭 系統架構總覽

```
勞工輸入 (對話 / 截圖 / 班表)
        │
        ▼
💬 AI Agent：法規比對 + 事實萃取 + 金額試算
        │
        ▼
🔗 區塊鏈存證：SHA-256 雜湊 + 模擬上鏈 (TxHash)
        │
        ▼
📄 自動生成勞資爭議調解申請書 (含鏈上證據編號)
        │
        ▼
🏛️ 模擬送交地方勞工局線上申訴系統
```

## 📖 平台使用說明 / Demo 劇情（5 分鐘）

平台內建「📖 使用說明」分頁，收錄以下完整 Demo 劇情，方便評審或使用者快速上手：

1. **勞工登入平台**：使用內建測試帳號（`labor01` / `test1234`）登入 Labor-Guard Agent。
2. **與 Agent 對話，描述狀況**：在「💬 AI 智慧蒐證對話」分頁輸入「公司未支付加班費」，Agent 開始引導蒐證。
3. **上傳關鍵證據**：上傳 LINE 對話、薪資單、打卡紀錄，系統自動計算每份檔案的 SHA-256 指紋。
4. **Agent 自動整理證據、辨識缺漏並比對法規**：Agent 解析上傳內容，自動比對《勞動基準法》相關條文，估算應補發金額。
5. **產生勞資爭議申請書草稿**：切換到「📄 調解申請與送件」分頁，系統依蒐證結果自動產出結構化申請書草稿。
6. **使用者確認內容並授權送件（Human Approval）**：勞工本人檢視草稿無誤後，按下「確認送出」按鈕。
7. **證據雜湊與操作紀錄寫入區塊鏈**：前往「🔗 區塊鏈證據鏈」分頁，一鍵上鏈並顯示交易雜湊（TxHash），驗證成功。
8. **在「我的案件」查看進度**：在「📊 我的案件」分頁查看目前送件與處理進度（四大階段時間軸）。

## 🛡️ Agent 治理設計：授權、控制與可追溯性

黑客松評審常關注的核心信任問題：**Agent 代表誰？被允許做什麼？高風險動作是否受控？行為是否可追溯？授權是否可撤銷？**
以下是 Labor-Guard Agent 針對這五個面向的治理設計。

### 一、授權邊界與身份對齊（Who authorized & What is allowed）

- **授權主體明確**：Agent 的所有行動皆綁定於明確登入的「勞工實名身份（數位憑證或帳號）」，並在對話及產出文件前，透過明確的授權同意書（Explicit Consent）確認：「本人授權 Labor-Guard Agent 於本次勞資爭議事件中代為蒐集、整理與格式化相關數位證據」。
- **權限最小化原則（Least Privilege）**：Agent 僅被賦予「讀取對話 / 出勤紀錄」、「計算加班費」、「草擬申請書」的權限，不具備直接代表勞工私自和解、放棄權利或執行具法律約束力之最終簽署的權利（高風險動作必須由使用者人工最終確認）。

### 二、高風險動作控制與防呆（High-Risk Action Control）

- **「人類在環」機制（Human-in-the-Loop）**：自動化 vs. 決策分級——證據解析與法規比對由 Agent 自動化完成；但「產生正式調解申請書」以及「最終送件至主管機關」，皆須經過勞工本人按下「確認送出」按鈕方能執行。
- **防幻覺與法規對齊護欄**：Agent 在輸出法規依據（如勞基法第 24 條）時，會強制掛載來源條文連結與檢核點，避免 AI 產生不實的法律見解導致勞工權益受損。

### 三、行為追溯與不可竄改性（Traceability & Immutability）

- **區塊鏈數位指紋（Blockchain Evidence Anchoring）**：當勞工上傳 LINE 截圖、出勤紀錄等敏感證據時，系統會立即計算其 SHA-256 數位雜湊值（Hash）。將該 Hash 上鏈（或記錄於可驗證的數位證據鏈結），產生唯一的 TxHash。
- **防造假保證**：即便未來資方質疑證據遭變造，勞工也能透過鏈上紀錄證明「此證據自蒐集當下起，內容絕對未被修改過」，完美解決勞資舉證不對等痛點。

### 四、授權撤銷與生命週期管理（Revocation & Lifecycle）

- **明確的效期與終止機制**：勞工與 Agent 的授權關係設定明確的生命週期（Lifecycle）。當勞資爭議案號結案、調解成立，或勞工手動點擊「撤銷授權與清除暫存資料（Revoke & Purge）」時，Agent 將立即終止存取權限。
- **隱私與資料權控**：所有暫存的對話與敏感截圖在任務結束或授權撤銷後，皆遵循資安規範進行生命週期管理，確保勞工隱私不外洩。

### 📊 治理設計對照總覽表

| 信任維度 | 傳統 AI 痛點 | Labor-Guard 治理對策 | 評審驗證點（Demo 呈現） |
|---|---|---|---|
| 誰授權（Who） | 誰發起指令模糊不清 | 綁定實名數位帳號與明確授權同意 | 登入頁面與授權勾選狀態 |
| 允許做什麼（What） | Agent 擁有過大自主權 | 權限分級：僅能蒐集與草擬，高風險動作需人工確認 | 申請書送出前的「二次確認機制」 |
| 如何追溯（Traceability） | 證據容易被質疑遭變造 | 檔案 SHA-256 雜湊上鏈，具備不可否認性（Non-repudiation） | 「區塊鏈證據鏈」分頁中的 TxHash 與驗證工具 |
| 如何撤銷（Revocation） | 授權無限期擴張、個資外洩風險 | 提供一鍵撤銷授權與生命週期終止機制 | 設定或狀態面板中的「終止 / 撤銷授權」按鈕 |

## 🚀 本機執行方式

```bash
# 1. 建立虛擬環境（可選）
python -m venv venv
source venv/bin/activate   # Windows: venv\Scripts\activate

# 2. 安裝套件
pip install -r requirements.txt

# 3. 啟動 Streamlit
streamlit run app.py
```

啟動後瀏覽器會自動開啟 `http://localhost:8501`。

## ☁️ 部署到 Streamlit Community Cloud

1. 將本專案上傳至 GitHub repository（含 `app.py`、`requirements.txt`）。
2. 前往 [share.streamlit.io](https://share.streamlit.io)，使用 GitHub 帳號登入。
3. 點選 **New app**，選擇你的 repository、分支，Main file path 填 `app.py`。
4. 點選 **Deploy**，即可取得公開網址分享給評審。

## 🛠️ 技術重點

- **前端 / 互動**：Streamlit（登入表單、側邊欄導覽 `st.sidebar.radio`、`chat_input`、`file_uploader`、`session_state`）
- **登入機制**：內建帳密字典模擬身份驗證（Demo 用，正式版可替換為公司 SSO / 數位憑證登入）
- **案件編號**：登入時以帳號 + 時間戳雜湊自動產生唯一案件編號（例如 `LDC-2026-8F3A2C`）
- **證據雜湊**：Python 內建 `hashlib.sha256`
- **區塊鏈上鏈**：以模擬 TxHash（可替換為實際串接 Polygon / Ethereum 測試網 SDK，如 web3.py）
- **法規比對**：規則式關鍵字比對（Demo 用，正式版可替換為 LLM / RAG 引擎串接勞基法資料庫）
